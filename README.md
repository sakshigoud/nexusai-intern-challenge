## Task 1 — AI Message Handler

**File:** `task1/handler.py`

An async function that sends a customer message to the OpenAI API and returns a structured `MessageResponse` with:
- `response_text` — AI-generated reply
- `confidence` — float between 0 and 1
- `suggested_action` — one of `resolve`, `escalate`, `follow_up`, `inform`
- `channel_formatted_response` — reply formatted for the channel (`voice`, `whatsapp`, `chat`)
- `error` — error message string if something went wrong, otherwise `None`

### How to Run

```bash
# Set your OpenAI API key (or add it to .env)
# Windows:
set OPENAI_API_KEY=sk-your-openai-api-key-here

# macOS/Linux:
export OPENAI_API_KEY=sk-your-openai-api-key-here

# Run the demo
python task1/handler.py
```

### Expected Output

```
Voice test: MessageResponse(
    response_text="I understand your frustration with the internet outage, and I sincerely apologize for the inconvenience. I'm escalating this to our technical team right now and you'll receive an update within 2 hours.",
    confidence=0.87,
    suggested_action='escalate',
    channel_formatted_response="I understand your frustration with the internet outage, and I sincerely apologize for the inconvenience. I'm escalating this to our technical team right now and you'll receive an update within 2 hours.",
    error=None
)
```

> **Note:** The exact `response_text` will vary because it is generated by GPT-3.5-turbo. The `confidence` score and `suggested_action` will also vary slightly between runs.

### Error Handling Behaviour

| Scenario | Result |
|---|---|
| Empty / whitespace input | `error="Empty or whitespace-only input received."`, `suggested_action="error"` |
| API timeout (> 10 s) | `error="API request timed out after 10 seconds."`, `suggested_action="escalate"` |
| Rate limit hit | Retries once after 2 s; if retry also fails, `suggested_action="escalate"` |

## Task 2 — Database Schema & Repository

**Files:** `task2/schema.sql`, `task2/repository.py`

### Step 1 — Create the table in PostgreSQL

```bash
psql -d your_database -f task2/schema.sql
```

This creates the `call_records` table with appropriate columns and indexes.

**What the SQL does:**
- Creates `call_records` table with fields: `id`, `customer_phone`, `channel`, `transcript`, `ai_response`, `intent`, `outcome`, `confidence_score`, `csat_score`, `duration_seconds`, `created_at`
- Adds B-tree index on `customer_phone` for fast lookups
- Adds B-tree index on `created_at` for time-range queries

### Step 2 — Use the Repository

The `CallRecordRepository` class provides three async methods:

| Method | Description |
|---|---|
| `save(call_data)` | Insert a call record, returns the new `id` |
| `get_recent(phone, limit=5)` | Get the N most recent records for a phone number |
| `get_low_resolution_intents(days=7)` | Top 5 intent types with the lowest resolution rate in the last N days |

**Example usage (requires a running PostgreSQL instance):**

```python
import asyncio
from task2.repository import CallRecordRepository

async def main():
    repo = await CallRecordRepository.create()  # reads DATABASE_URL from .env

    # Save a call record
    record_id = await repo.save({
        "customer_phone": "+1-555-0100",
        "channel": "chat",
        "transcript": "My bill is wrong.",
        "ai_response": "Let me look into that for you.",
        "intent": "billing_dispute",
        "outcome": "resolved",
        "confidence_score": 0.91,
        "csat_score": 4,
        "duration_seconds": 120,
    })
    print(f"Saved record ID: {record_id}")

    # Fetch recent records
    records = await repo.get_recent("+1-555-0100")
    print(records)

asyncio.run(main())
```

**Expected output:**
```
Saved record ID: 1
[{'id': 1, 'customer_phone': '+1-555-0100', 'channel': 'chat', ...}]
```

## Task 3 — Parallel Data Fetcher

**File:** `task3/fetcher.py`

Demonstrates the performance improvement of `asyncio.gather` over sequential `await` calls by fetching mock customer data from three simulated systems (CRM, Billing, Tickets) with realistic random delays.

**No external dependencies or API keys required.**

### How to Run

```bash
python task3/fetcher.py
```

### Expected Output

```
==================================================
Sequential Fetch
==================================================
  Time: 723.45 ms
  Data complete: True

==================================================
Parallel Fetch
==================================================
  Time: 389.12 ms
  Data complete: True

Speedup: 1.86x
```

> **Note:** Exact timing values will vary between runs due to random delays. Values outside the ranges below indicate something unexpected.

### Expected Timing Ranges

| Fetch Mode | Expected Time | Explanation |
|---|---|---|
| **Sequential** | 450 – 1 050 ms | Sum of all three delays (CRM + Billing + Tickets) |
| **Parallel** | 200 – 400 ms | Only as long as the slowest single fetch |
| **Speedup** | ~1.5x – 3x | Typically close to **2x** |

### Billing Failure Handling

The billing mock has a **10% chance of timing out**. When it fails:
- A warning is logged: `WARNING:__main__:Billing fetch failed: Billing system timed out`
- `data_complete` is set to `False`
- The other two data sources are still returned

## Task 4 — Escalation Decision Engine

**Files:** `task4/escalation.py`, `task4/test_escalation.py`

A pure Python function `should_escalate()` that decides whether a customer interaction should be handled by AI or routed to a human agent. It uses **6 rules checked in priority order** (first match wins).

### Escalation Rules

| Priority | Rule | Trigger | Reason Returned |
|---|---|---|---|
| 1 | Low confidence | `confidence_score < 0.65` | `"low_confidence"` |
| 2 | Angry customer | `sentiment_score < -0.6` | `"angry_customer"` |
| 3 | Repeat complaint | Same intent appears 3+ times in ticket history | `"repeat_complaint"` |
| 4 | Service cancellation | `intent == "service_cancellation"` | `"service_cancellation"` |
| 5 | VIP billing issue | Customer is VIP **and** billing is overdue | `"vip_billing_issue"` |
| 6 | Incomplete data | `data_complete == False` and `confidence_score < 0.80` | `"incomplete_data"` |

### How to Run Tests

```bash
pytest task4/ -v
```

### Expected Test Output

```
==================== test session starts ====================
platform win32 -- Python 3.x.x, pytest-x.x.x
collected 8 items

task4/test_escalation.py::test_rule1_low_confidence           PASSED
task4/test_escalation.py::test_rule2_angry_customer           PASSED
task4/test_escalation.py::test_rule3_repeat_complaint         PASSED
task4/test_escalation.py::test_rule4_service_cancellation     PASSED
task4/test_escalation.py::test_rule5_vip_billing_overdue      PASSED
task4/test_escalation.py::test_rule6_incomplete_data_low_confidence PASSED

```

All **8 tests must pass**.

### Rule Conflict Note

When multiple rules could apply to the same input, the engine picks the **first matching rule** in priority order. For example, a `service_cancellation` intent with `confidence=0.99` will trigger Rule 4 — not a "safe" pass-through — because service cancellations are always high-stakes and require human handling.
